第八周

发表于 2016-11-20   |   分类于 每周总结   |  

  这周主要做了两件事：

  1. 过了6道leetcode
  2. 开题方面，实现了之前的一个优化的小想法。

    开题方面

      目前只在一个数据集上做了测试。因为在大图上做实验，自己的机子的内存吃不消。以前就出现过OutOfMemory错误，而实验节点最近出故障了。但是在现在的步长参数下，得到的mean_mirror是已有算法的一半左右，应该来说还是有效果的。只是一个很小的想法，也很容易想到，没有相应的模型和理论支撑，慢慢来吧。
      另外的体会就是，自己对maven还是停留在能用阶段，对java的打包机制也不熟。因为要用解析一个数据集，需要相应的包。将从github上maven项目导入IntelliJ后可以用，但是用命令行执行就报错can’t load main class…，不晓得。
      另外对内存GC机制，是一个盲区。
      最后就是java也提供文件内存映射相应的api，接下来会考虑用MappedByteBuffer读写文件，这样占用的内存小些。

    leetcode

      做了六道题，几点体会如下：
    1.#119，这里涉及到迭代覆盖的问题，从后往前a[j]=a[j]+a[j-1]，迭代完后再最后补上第一次被覆盖的元素，从前往后也是可以的。
    2.#448和#442是一个思路，题目关键是1<=a[i]<=n，n是数组的长度，这样就可以考虑一个两次映射用a[a[i-1]]的正负来标记a[i]是否已出现过。
      以上问题的难点都在于在受限的空间内完成，解决办法一个是覆盖，另一个是对元素加可还原标记。

近期总结

发表于 2016-11-08   |   分类于 每周总结   |  

  一个月没更新了。很多破事不说了。自己在开题方向上也是乱撞。还好有点方向了，落下的leetcode补了25题。
  最近用了IntelliJ写Java，调试方便、自动显示类信息，体验很好。不知道为什么以前一直只用编辑器，很多时候找个类找半天，调试更坑，很影响情绪。以前中毒很深的。还是要实际点，不要急功近利。
  最近一段时间计划是这样的，重新开题最晚年底，确保开题顺利通过。在开题结束之前，白天实现自己的想法，不断修改做实验。晚上做题，尽量一天一题。开题通过后，就可以多一些自己的时间了。
  最近做题还是有收获的，AC后看一下discuss里的高票答案收获很大。同时还有相应题目的article（这个目前还没看）。毕竟一个人闷着头做，知道的技巧是很有闲的。学习到的技巧可以用在以后的做题上。总结如下：

  1. #384使用异或运算符找两个字符串的不同字符。
  2. #414维持一个滑动窗口，找前三大的元素
  3. #62将递归方法转换成迭代的方法，递归超时。
  4. #46输出给定字符串的所有排列，即使是递归解都是不好想的，理解也需要花一些时间。

  这些问题也有最容易想到的方法，但是时间复杂度都是比较高的，而且思路上没什么特别的。每个题目都值得多研究下，然而时间有限，先过数量吧。另外很多题目，隔得时间稍微长就忘了，这个总结很不到位。今后还是尽量一个星期总结一次，印象深一点。
  开题方面，之前的想法已经初步实现了。计划将几个已有的算法改进，实现一个单线程版本，在开题前有初步的数据。然后尝试实现一个多线程的应用。最后如果尝试将算法整合到具体的框架上。
  走了好多弯路，前段时间想着在直接在Giraph上实现想法，环境也搭好了，demo也可以跑，结合文档，代码也看了一些，没找到相应的api支持，然后又转Flink…
  直接想在框架上实现算法可能不切实际吧，源码论斤算,也不是好看的。还是实际点，一步步来。
  之前用的Next主题崩了,换了一个。

第二周

发表于 2016-10-02   |   分类于 每周总结   |  

  这周主要做了两件事：

  1. 过了6道leetcode题目
  2. 基于flask实现一个TODO List应用（尚未完工）

  感觉leetcode上有些题目还是挺难的，做的题太少，很多技巧不知道。直接硬想，想半天只能模模糊糊有点思路，到细节的地方又卡壳了。对于跟树有关的题目，之前一直觉得非递归解决问题才是最好的，然而很多问题非递归解法是很难的，（比如leetcode#113）网上能找到的都很少。所以能解决问题就是最好的，一步步来吧，不要一开始就要用技巧性最强的解法。
  leetcode#113是跟树遍历相关的题目，最开始想的是借鉴后续遍历的想法去做，所以又复习了一遍树的前中后序遍历的递归和非递归方法，然而想偏了。
  最后总结下做完这6道题的收获:

  1. leetcode#11, 考虑前后两个指针的思路，这种方式应该有通用性。
  2. leetcode#113, 考虑递归的时候，考虑第一次递归和最后一次递归（根节点和叶子节点）便于理解。

  最后初步理解了ORM框架是什么。
